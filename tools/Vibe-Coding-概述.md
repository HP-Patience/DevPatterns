# 从 Vibe Coding 到 Agentic Engineering

> 2025 年初，Andrej Karpathy 提出 **Vibe Coding**——开发者靠自然语言描述需求，"Accept All"，不读代码差异，让 AI 全权产出。这个概念的流行说明了一个事实：AI 写代码已经不是能不能的问题，而是怎么用的问题。
>
> 2026 年 2 月，Karpathy 亲自宣告 Vibe Coding 过时，提出替代概念 **Agentic Engineering**。不是全盘否定，而是进化——从"放手让 AI 写"到"指挥和监督 AI 写"。

---

## 一、Vibe Coding 是什么（历史回顾）

Vibe Coding 的核心："你说要什么，AI 写好，你只管看结果。"

工作流：描述需求 → AI 生成 → 看结果 → 给反馈 → AI 调整 → 循环

人类角色从"编码员"变成**出题人 + 审核人**。

### 它当年为什么火

| 方面 | 传统开发 | Vibe Coding |
|------|---------|-------------|
| 速度 | 搭环境到出原型按天算 | 按分钟算 |
| 门槛 | 会语言/框架/调试 | 会描述需求就行 |
| 体验 | 卡在细节，创意被打断 | 专注"要什么" |
| 试错成本 | 改方向要重写 | 改描述就行 |

核心价值在于证明了：**AI 可以承担实现层，人类应该聚焦于判断和决策**。

---

## 二、Vibe Coding 在实践中暴露的问题

"Accept All" 模式在真实项目里翻车的点：

1. **迭代混乱** — 改一个功能，AI 顺手改歪三个其他模块。频繁回滚，越修越糟
2. **Prompt 丢失** — 写出好结果时的关键提示词没保存，回头想优化找不到入口
3. **安全漏洞** — AI 不会主动防 SQL 注入、XSS、权限泄露，全盘接收等于请黑客做 code review
4. **技术债泛滥** — AI 倾向于复制粘贴式扩展，代码量膨胀快，可维护性下降更快
5. **无差异审查** — 不看代码直接合入，等于把工程判断权完全交出去

一句话总结：**Vibe Coding 快，但不稳。适合一次性原型，不适合持续迭代的项目。**

---

## 三、Agentic Engineering：Karpathy 的下一站

2026 年 2 月 4 日，Karpathy 正式提出 **Agentic Engineering**，核心变化：

| 维度 | Vibe Coding | Agentic Engineering |
|------|------------|-------------------|
| 开发者角色 | 出题人，Accept All | 指挥官 + 监督者，审阅每份产出 |
| AI 角色 | 自动代码生成器 | 可编排的 Agent，执行-反馈-优化 |
| 工作方式 | 对话式，一次一题 | 系统化，多 Agent 协作 + 流程管控 |
| 代码审查 | 不看差异 | 逐行 Review，理解每处修改 |
| Prompt 管理 | 即用即丢 | 结构化 Spec 文档，可复用可追溯 |
| 工程标准 | 无 | 安全、性能、可维护性纳入衡量 |

### Agentic Engineering 的实践形态

不是回到手写代码时代，而是升级人跟 AI 的协作方式：

**不再"聊天写代码"，而是"指挥 Agent 做工程"。**

实际做起来大概是：

```
1. 写 Spec — 结构化需求文档，明确功能边界和验收标准
2. 指派 Agent — 让 AI Agent 按 Spec 执行，而不是自由发挥
3. 审查产出 — 逐 Diff 过代码，理解每一处变化的理由
4. 反馈迭代 — 不合格的打回，合格的合入
5. 沉淀资产 — 好的 Prompt、Spec 模板、Agent 配置存档复用
```

这是 Spec Coding 的核心思路：**先写清楚要什么，再让 AI 去做，而不是边聊边做。**

## 四、对你意味着什么

工具没变——Cursor、Claude、Cline 这些还是那些工具。变了的是用法：

| 旧习惯 | 新习惯 |
|--------|--------|
| "帮我写个登录页面" | "按 spec.md 第三节的登录方案实现" |
| "这里改一下" | "修改点：缩小间距 8px，颜色改成 tokens.color.primary" |
| Accept All | 逐段 Review，理解再合入 |
| 用完即忘的 Prompt | 结构化 Spec 文档，反复复用 |
| AI 写啥就啥 | 设定工程标准，不达标打回重做 |

**核心转变：从"让 AI 写代码"到"管理 AI 写代码"。**

你的价值不再是你打字多快、记多少 API，而是：

- 能不能写清楚一份 AI 能执行的 Spec
- 能不能判断 AI 生成的代码够不够好
- 能不能设计 Agent 工作流让 AI 少犯错

## 六、一句话总结

**Vibe Coding 证明了 AI 能写代码，Agentic Engineering 告诉我们怎么让 AI 写好代码。**

前者回答了"能不能"，后者回答"怎么管"。真正有用的不是谁来写，而是谁来把关。
