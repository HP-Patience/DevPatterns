# Vibe Coder 视角的 Handoff：你需要吗？怎么用？

> 看完手写笔记那个视频，你可能在想："我一人做项目，又不搞多 agent 协作，handoff 跟我有什么关系？"
>
> 直说答案：**你大概率不需要 handoff 给别人，但你非常需要 handoff 给自己。**

---

## 一、Vibe Coder 的典型会话轨迹

```
第 1 天：激情开干，让 AI 搭落地页首屏 ← 聪明区
第 2 天：加功能、调样式、换素材
第 3 天：改 Bug、优化、再加功能 ← 开始迟钝
第 4 天：AI 开始忘上下文、生成重复代码 ← 迟钝区
第 5 天：受不了了，新开会话从头再来 ❌
```

这跟上下文窗口大小无关。就算给你 100 万 token，**内容越多注意力越分散，输出质量越差**。

你的问题是：**一个好端端的项目，被一个越来越臃肿的会话拖死。**

---

## 二、Handoff 对 Vibe Coder 的真正价值

不是跨 agent 协作，而是**给自己做任务隔离**。

### 场景 1：做落地页时发现后端接口没写好

```
❌ 坏做法：在同一个会话里让 AI 先写接口，再继续做页面
    → 上下文被后端代码污染，前端生成质量下降

✅ Handoff：让当前会话写一份 handoff 文档
    → 新会话专门处理后端接口
    → 原会话继续纯前端工作，两个都干净
```

### 场景 2：做完一期功能，想做二期

```
❌ 坏做法：直接在原会话继续加需求
    → 上下文越来越重，AI 开始胡搞

✅ Handoff：结束一期时让 AI 写项目状态总结
    → 新会话从干净状态开始二期
    → 带上一期的关键上下文，不带垃圾
```

### 场景 3：AI 卡在某问题上反复绕圈

```
❌ 坏做法：不断"继续"让 AI 重试
    → 上下文全是失败尝试，越绕越深

✅ Handoff：让当前会话总结"已尝试的方案和为什么失败"
    → 新会话带着失败记录，从干净上下文重新思考
    → 经常一次就找到正确方向
```

## 三、给你的轻量 Handoff 模板

### 什么时候 handoff

当出现以下信号之一：

- 当前任务**明显超出**本会话原始目标
- 会话已超过 **100-150 条消息**，感受 AI 变笨
- AI 开始**重复已解决的问题**或忘记上下文
- 需要**暂停主线**去做另一个独立任务

### Handoff 怎么写

直接在当前会话说：

```
写一份 handoff 文档，用于交给新会话处理 [具体任务]。

内容包括：
1. 任务目标 — 新会话要做什么
2. 已完成的相关工作 — 哪些已经做了，避免重复
3. 关键上下文 — 文件路径、技术选型、约束条件
4. 已知陷阱 — 踩过的坑、试过不行的方案
5. 建议技能 — 新会话应该用哪些技能

存到 OS 临时目录，不要放项目里。
```

### 收到 handoff 后怎么做

```
已收到 handoff 文档。让我先读完，然后开始执行 [任务目标]。

执行中会保持：
- 不偏离 handoff 定义的范围
- 完成后写结论，不清扫原会话的垃圾
- 如果发现 handoff 中未覆盖的问题，记录下来
```

## 四、什么时候不需要 handoff

| 情况 | 判断 |
|------|------|
| 一次性小修改（改个文案、换个颜色） | 直接干，别折腾 |
| 会话刚开始，AI 还很聪明 | 继续用 |
| 纯粹浏览代码/查资料 | 不需要 |
| 项目只有 1-2 个页面，一个人几天做完 | 不需要 |

Handoff 是**解决复杂度**的工具，不是**制造复杂度**的工具。等真正感觉到"这个会话太重了"再用。

---

## 五、再说回 Compact（压缩）

Compact 和 Handoff 不是二选一：

```
Compact = 续杯。同一件事做到底，不断压缩重来。
Handoff = 分杯。把一件事切成多块，并行或分阶段做。

Vibe Coder 最实用的组合：
- 一个项目拆成 N 个手写文档（功能模块）
- 每个模块一个会话，各自 compact 推进
- 模块间通过 handoff 交换关键上下文
```

不需要懂 agent、不需要懂多会话架构。只需要知道：**一个会话只做一件事，做完就归档，下个会话从干净状态开始。**
